# 幻觉

> 理解 AI

AI 为什么会一本正经地胡说八道？

你可能遇到过这样的场景：让 AI 帮你查一篇论文，它给了你一个看起来很正规的标题、作者和期刊 — 但你去搜，发现这篇论文根本不存在。或者让它写一段调用后端接口的代码，字段名、返回格式看着都对，但实际接口根本不是这么定义的。

AI 没有在「骗你」，它只是在做它最擅长的事：**根据已有信息，生成最像正确答案的文本**。当信息不够时，它不会说「我不知道」，而是会自信地「编」一个出来。

这种现象，就是 **幻觉（Hallucination）**。

理解幻觉是什么、为什么会产生、怎么应对，是用好 AI 的必修课。否则你很容易被一段「看起来很专业」的输出误导，而浪费大量时间在一个根本不存在的方向上。

---

## 一、幻觉是什么

### 1. 定义

幻觉是指大模型生成的内容看起来流畅合理，但实际上与事实不符、与给定的上下文矛盾，或者根本无法被验证为真。

常见的表现形式：

| 类型 | 表现 | 例子 |
|------|------|------|
| **捏造事实** | 编造不存在的信息 | 虚构论文标题、伪造 API 文档链接、编造不存在的库函数 |
| **与输入矛盾** | 无视用户提供的信息 | 你说用 Swift，它给你写 Kotlin；你给了接口定义，它用了不同的字段名 |
| **过度补全** | 在信息不足时凭空填充 | 你只给了部分表结构，它「猜」出了其余字段的名称和类型 |
| **张冠李戴** | 把不相关的知识混在一起 | 把 A 框架的用法套在 B 框架上，语法对但运行报错 |

在日常编程中，幻觉最典型的表现是：**代码看起来能跑，但接口不存在、变量名对不上、逻辑与现有架构不一致**。这类错误比明显的语法错误更危险，因为它不容易被一眼发现。

### 2. 幻觉与创造力：同一枚硬币的两面

这是理解幻觉最关键的认知：**幻觉和创造力来自同一种机制**。

模型之所以能帮你补全代码、写文档、推断需求、生成方案，正是因为它会在信息不完整时做推测和联想。这种能力用对了，就是创造力；偏了，就是幻觉。

| | 创造力 | 幻觉 |
|---|---|---|
| **机制** | 基于已有信息做推测和补全 | 同左 |
| **结果** | 推测符合事实或上下文 | 推测偏离事实或上下文 |
| **感受** | 「AI 真聪明，这都能想到」 | 「AI 在瞎编，这根本不对」 |

所以业界常说：**幻觉不是 bug，而是模型工作方式带来的特性**。你不能「关掉」幻觉而保留创造力，因为它们是一体的。真正的应对思路是：**保留模型的推测能力，同时用上下文、工具和规则把它的输出约束在可控范围内**。

## 二、幻觉为什么会产生

从三个层面来理解，由浅到深。

### 第一层：模型看到的信息永远是不完整的

任何大模型都有 **Context Window（上下文窗口）** 的限制 — 它一次能「看到」的信息量是有上限的（目前主流模型在 128K ~ 200K Token 之间）。

但真实的项目往往包含：

- 几十万行代码
- 多年积累的需求文档和变更历史
- 架构设计、API 文档、数据库定义、部署配置……

这些信息不可能每次都全部塞进上下文。当模型只能看到局部信息时，它就会基于训练数据和已有上下文做「合理推测」。推测对了，你觉得 AI 很懂；推测偏了，就成了幻觉。

**根本原因：上下文有界 → 模型始终在「信息不完整」的条件下工作 → 推测是必然的，偏差也是必然的。**

### 第二层：模型的工作方式决定了它「不会说不知道」

回顾 [Token](../token/) 那篇的核心结论：大模型本质是概率预测 — 给定前文，预测最可能的下一个 Token。

这意味着：

- **信息充足时**：预测大概率贴合事实，输出靠谱
- **信息不足时**：模型不会停下来说「这个我不确定」，而是继续预测一个「看起来最像正确答案」的输出

比如：

> 你让 AI 调用一个内部接口，但没给接口文档。AI 不会回你「我不知道这个接口的字段」，而是会根据接口名称和上下文「猜」一组字段名和类型 — 猜对了是能力，猜错了就是幻觉。

这就是为什么幻觉往往很「自信」— 模型从概率角度认为自己的输出是合理的，它没有「不确定」的概念。

### 第三层：这不是缺陷，而是设计取舍

假设我们强制要求模型：有信息就输出，没信息就沉默。那它就退化成了一个严格的检索系统 — 只能回答训练数据里明确存在的内容。

这样做的代价是：

- 代码补全？做不了，因为你的代码不在训练数据里
- 需求推断？做不了，因为没有完全匹配的先例
- 方案设计？做不了，因为没有和你项目一模一样的案例

模型之所以有用，恰恰是因为它敢于在信息不完整时做推测。**幻觉是创造力的副作用，不是系统错误。** 应对的思路不是「消除推测」，而是「让推测有据可依」。

## 三、如何应对幻觉

既然幻觉不能被消除，那就需要在实践中把它控制在「可发现、可纠正」的范围内。以下四个策略，从源头到终端层层递进。

### 1. 提供充足且结构化的上下文

幻觉的首要诱因是信息不足。所以最直接的应对就是：**把模型需要的信息给够**。

**给准范围** — 不是把所有文件都塞进去，而是只放与当前任务直接相关的内容：当前文件、依赖的模块、相关的接口定义、必要的约束说明。

**结构化组织** — 用清晰的标题、列表、代码块组织信息，让模型能快速「看到」关键约束。一段排版混乱的长文本，模型很容易遗漏关键信息。

**标注边界** — 明确告诉模型哪些是「必须遵守的」（如接口字段定义、命名规范），哪些是「仅供参考的」。边界越清晰，模型自由发挥的空间越小，幻觉越少。

> 目标：让模型在信息相对完整的局部上工作，而不是在全局信息缺失下猜测。

### 2. 用工具替代「猜」

很多幻觉的本质是：模型在「应该查数据」的地方「凭记忆编」。解决方法就是让模型去查，而不是去猜。

**能查就查** — 用检索、读文件、调 API 等方式获取真实数据，再让模型基于查到的结果生成内容。比如让模型先读取接口定义文件，再写调用代码，而不是让它凭空编造字段。

**工具调用（Tool Calling / MCP）** — 通过标准协议让模型连接代码库索引、文档库、数据库等外部数据源。这样模型的输出有了真实数据的支撑，从源头减少捏造。

> 目标：用真实数据约束输出，而不是依赖模型的记忆或推测。

### 3. 用规则和格式约束输出

即使信息给够了，模型仍然可能在细节上「创造性发挥」。通过规则和格式约束，可以进一步收窄它的输出范围。

**约定输出格式** — 要求模型在回答中引用文件路径、列出依据来源、标注不确定的部分。这样即使有幻觉，也容易被发现。

**写明规则和规范** — 在 Prompt 或项目配置中明确命名规范、架构约束、禁止模式。比如「所有接口调用必须基于 api.ts 中的定义，不要自行编造字段」。

**分步确认** — 对关键操作（如修改接口、删除逻辑、改数据库结构），要求模型先给出计划，人工确认后再执行。避免模型一口气做完，幻觉扩散到多个文件后才被发现。

> 目标：把幻觉控制在可发现、可纠正的范围内。

### 4. 人工与自动化校验兜底

前三个策略是「预防」，这一步是「兜底」— 不依赖「模型永远正确」，而是用流程来发现和修正错误。

**关键结果必查** — 对模型生成的接口调用、配置项、数据库字段等做人工抽查或脚本校验。越是「看起来合理」的输出，越要多看一眼。

**测试验证** — 用单测、接口测试、回归用例验证模型改过的部分是否符合预期。很多幻觉在编译时不报错，但在运行时会暴露。

**迭代修正** — 把模型输出视为初稿而不是终稿。通过 Code Review、测试结果、实际运行反馈形成闭环，逐步修正问题。

> 目标：建立「发现 → 修正 → 反馈」的闭环，而不是一次性信任模型的输出。

## 四、小结

| 维度 | 要点 |
|------|------|
| **幻觉是什么** | 模型生成的看似合理但与事实不符的内容，包括捏造事实、与输入矛盾、过度补全 |
| **为什么产生** | 上下文有限 + 概率预测机制 → 信息不足时模型会「合理猜测」而非拒绝回答 |
| **与创造力的关系** | 同一机制的两面：推测对了是创造力，偏了是幻觉，不能只关一个 |
| **应对策略** | ① 给够上下文 → ② 用工具替代猜测 → ③ 用规则约束输出 → ④ 人工和测试兜底 |
| **核心心态** | 把模型输出视为「高质量初稿」而非「最终答案」，用流程而非信任来保证质量 |

一句话总结：**大模型在信息不足时会主动「合理猜测」，这既是它有用的根本原因，也是幻觉的根本来源。通过更好的上下文、工具调用和规则约束，可以在保留创造力的同时，把幻觉控制在可接受、可纠正的范围内。**
