# MCP

> 用好 AI

MCP（Model Context Protocol，模型上下文协议）让 AI 能连接外部系统——查数据库、调 API、读写文件、操作工具。没有它，AI 只能给建议；有了它，AI 能真动手。

---

## 一、为什么需要 MCP

AI 本质上是一个封闭系统：你给它文本，它返回文本。它看不到你的文件、连不上数据库、调不了 API。

之前几篇讲的 [Prompt](../prompt/)、[Rules](../rules/)、[Skills](../skills/) 都在解决「AI 怎么思考和执行」的问题。但如果 AI 没有手去触达真实世界，再好的思考也只能停留在纸面上。

MCP 就是给 AI 装上手的标准协议。

### 没有 MCP 之前

每对接一个外部工具，就要写一套适配代码：

```
接 GitHub → 写一套
接 Slack  → 再写一套
接数据库  → 再写一套
换一个 AI 工具 → 全部重写
```

M 个 AI 应用 × N 个外部工具 = **M × N 个适配器**，工作量随工具数量爆炸增长。

### 有了 MCP 之后

MCP 定义了一个统一协议：工具端实现一次 MCP Server，就能被所有支持 MCP 的 AI 调用；AI 端实现一次 MCP Client，就能接入所有 MCP Server。**M × N 变成了 M + N**。

这和 USB 的故事一样——USB 出现之前，每个外设有自己的接口；USB 出现之后，一个标准走天下。MCP 就是 AI 世界的 USB。

## 二、MCP 的架构

### 三个角色

| 角色 | 作用 | 示例 |
|------|------|------|
| **Host（宿主）** | 面向用户的 AI 应用 | Claude Desktop、Cursor、Claude Code |
| **Client（客户端）** | Host 内部负责通信的模块 | 自动管理，用户无需关心 |
| **Server（服务端）** | 工具和数据源的提供方 | GitHub Server、数据库 Server、Figma Server |

### 一次调用的流程

以「查询最近 7 天活跃用户数」为例：

```
你说："查一下最近 7 天的活跃用户数"
    ↓
AI 理解意图，决定需要查询数据库
    ↓
AI 通过 MCP 协议调用数据库 Server
    ↓
Server 执行 SQL 查询，返回结果
    ↓
AI 基于结果生成回答："最近 7 天活跃用户 12,483 人，环比增长 8.2%"
```

### Server 提供的三种能力

| 能力 | 说明 | 控制方 | 示例 |
|------|------|--------|------|
| **Tools（工具）** | 可执行的函数 | AI 模型 | 搜索航班、发送消息、创建日历事件 |
| **Resources（资源）** | 只读数据源 | 应用程序 | 文件内容、数据库记录、API 响应 |
| **Prompts（模板）** | 预制的 Prompt 模板 | 用户 | 计划旅行、总结会议、起草邮件 |

最常用的是 **Tools**——AI 判断需要执行某个操作时，调用对应的 Tool，拿到结果后继续推理。

### 两种通信方式

| 模式 | 通信方式 | 适用场景 |
|------|---------|---------|
| **Stdio** | 本地进程间通信（stdin/stdout） | 本地工具：文件操作、命令执行 |
| **HTTP** | 网络请求 | 远程服务：云端 API、跨机器调用 |

### 工具描述决定调用质量

AI 完全依赖 Tool 的名称、描述和参数定义来决定什么时候调用、怎么传参。描述写得不好，AI 要么不调用，要么传错参数。

| 写法 | 效果 |
|------|------|
| ❌ `name: "query"`, `description: "查询数据"` | AI 不知道查什么数据、参数传什么 |
| ✅ `name: "search_documents"`, `description: "在知识库中搜索文档，返回最相关的前 5 条结果"` | AI 能准确理解能力边界和使用场景 |

核心原则：**把工具描述当作给 AI 看的 API 文档来写**——名称有语义，描述说清功能边界，参数给出示例值。

## 三、MCP 的安全机制

AI 能调用外部工具了，怎么保证它不乱来？

| 机制 | 说明 |
|------|------|
| **权限控制** | 可限制 Server 只暴露特定 Tools（如只允许读，不允许写） |
| **操作确认** | AI 调用工具前可要求用户确认，而非静默执行 |
| **操作审计** | 所有工具调用都有记录，可追溯谁在什么时候调了什么 |
| **沙箱隔离** | MCP Server 作为独立进程运行，与 AI 应用隔离 |

## 四、常见的 MCP Server

社区已经有大量现成的 MCP Server，覆盖常见的开发场景：

| MCP Server | 能力 | 典型场景 |
|-----------|------|---------|
| **figma** | 读取设计稿结构化数据 | 设计转代码 |
| **filesystem** | 读写本地文件 | 读取配置、生成代码文件 |
| **github** | 操作 GitHub 仓库 | 创建 PR、查看 Issue、读取代码 |
| **postgres / mysql** | 数据库查询 | 数据分析、表结构查看 |
| **browser** | 浏览器自动化 | 网页抓取、前端测试 |
| **sentry** | 错误监控 | 崩溃分析、日志查询 |
| **slack / 飞书** | 消息通知 | 发送通知、查询聊天记录 |

不需要自己写这些 Server，直接配置即可使用。

查找更多：[MCP 官方 Server 仓库](https://github.com/modelcontextprotocol/servers)

## 五、MCP、Function Calling、CLI 的分层关系

这三者容易混淆，因为表面动作很像——都是「AI 调工具」。但它们处于不同的层次，解决不同的问题：

| 概念 | 层次 | 解决的问题 |
|------|------|-----------|
| **Function Calling** | 模型层 | 模型怎么表达「我要调用某个工具」的意图 |
| **MCP** | 接入层 | 工具怎么被统一注册、发现、接入 Agent 系统 |
| **Skills** | 规划层 | 任务怎么被拆解成可执行的步骤序列 |
| **CLI** | 执行层 | 这些步骤最终以什么动作形态被执行 |

### 完整链路示例

用户说：「帮我查一下张三上周的报销总额」

**第一层：Function Calling（模型表达意图）**

```json
{
  "name": "reimbursement_query",
  "arguments": { "employee_name": "张三", "period": "上周" }
}
```

**第二层：MCP（宿主找到对应工具）**

宿主程序在已接入的 MCP Server 中找到 `reimbursement_query`，把参数转发过去。

**第三层：Skills（规划执行路径）**

Skill 把「查张三上周报销」拆成一条完整路径：先获取员工 ID → 解析时间范围 → 查询报销记录 → 按项目汇总 → 输出结果。

**第四层：CLI（实际执行动作）**

```bash
employee get --name 张三
time parse --text 上周
reimbursement query --employee-id 123 --start 2025-12-08 --end 2025-12-14
reimbursement summarize --group-by project
```

四者的关系：Function Calling 让模型说出要做什么，MCP 让工具被找到，Skills 把任务规划成步骤，CLI 把每个步骤变成可执行的动作。

## 六、CLI 为什么越来越重要

### CLI 不是命令行界面，是能力暴露方式

很多人一听 CLI，想到的是黑窗口。但在 Agent 语境下，CLI 的本质是：**把系统能力结构化、参数化、变成可调用的标准动作**。

GUI 里的每次点击，背后都对应一个具体动作。CLI 做的事情，是把这些动作从界面里剥离出来，变成可以被人、脚本、Agent 直接调用的标准接口：

```bash
# GUI 里点 3 层菜单才能找到的功能
# CLI 一行命令直达
dingtalk doc-space usage
dingtalk doc-space upgrade --plan pro_100g --dry-run
```

### GUI 的真实困境

GUI 对人类本来很友好——信息量大，指哪打哪。但问题是平台给的太多了：

> 我们可能只使用了系统不到 20% 的功能，剩下 80% 被埋在层层菜单里。想关个京东白条？得搜攻略。想升级钉钉文档空间？在后台迷宫里转圈。

CLI 的价值就在这里——它能帮 Agent 触达那 80% 被深埋的功能。对 Agent 来说，一条命令直达，不需要像人一样在界面里翻找。

### 为什么钉钉、飞书在做 CLI 改造

这些平台正在把 GUI 背后的能力重新暴露成 CLI 形式，不是要做一个「黑窗口版钉钉」，而是让 Agent 能直接调用原本埋在 GUI 深处的能力：

```bash
# 查询类
dingtalk doc-space usage
dingtalk doc-space plans list

# 操作类
dingtalk doc-space upgrade --plan pro_100g --confirm

# 权限类
dingtalk doc-space upgrade permission-check
```

核心动机：**当执行者从「只有人类」变成「人类 + Agent」时，系统能力需要以 Agent 友好的方式暴露出来。**

### CLI 天然适配 Skills

Skills 围绕「动作」来组织 SOP：先做什么、再做什么、失败了怎么办。而 CLI 的世界里，一个命令就是一个动作、一组参数就是动作边界。两者天然契合：

```
# Skill 定义的执行路径
1. 查询当前空间使用情况
2. 查询可升级套餐
3. 检查用户权限
4. 有权限 → 执行升级（先 dry-run）
5. 没权限 → 查管理员，发起审批
```

每一步都能直接对应一条 CLI 命令。CLI 没有解决规划问题，但它让规划特别容易落地。

### Agent 怎么知道 CLI 的用法

两种模式：

**模式一：每个 CLI 命令包装成独立 Tool**

宿主把每条 CLI 命令定义成结构化的 Function Calling tool，模型看到的是工具名 + 参数 schema，不需要知道底层是 CLI。稳定但接入成本高。

**模式二：只暴露一个 `shell` 超级工具**

宿主只暴露一个通用执行入口：

```json
{ "name": "shell", "description": "执行 shell 命令" }
```

模型通过 `--help` 在运行时探索具体命令的参数结构：

```bash
dingtalk doc-space upgrade --help
# 输出参数说明后，模型再组织正式调用
dingtalk doc-space upgrade --plan pro_10g --dry-run --output json
```

灵活但稳定性不如第一种，且安全边界更模糊。

## 七、MCP vs CLI：不是替代，是分工

### 核心差异

| 维度 | CLI | MCP |
|------|-----|-----|
| **Token 消耗** | 极低（只传一个 bash 命令） | 高（所有工具描述都要传入上下文） |
| **执行效率** | 高（管道组合，一次调用完成） | 低（多轮调度，每步经过 AI） |
| **灵活性** | 最高（管道符自由组合） | 受限于工具设计者的预设 |
| **可控性** | 低（特殊字符易出错） | 高（JSON 结构化传参） |
| **安全性** | 低（命令无边界） | 高（只能执行允许的操作） |

### Token 消耗的真实差距

以 GitHub MCP Server 为例：44 个工具，全部描述约 163,700 字符，约 14,268 个 Token。按 Claude Sonnet 定价约 3 美分/次。同时启用多个 MCP Server，仅工具说明就要消耗可观的 Token。

CLI 方式只需一个 `bash` 工具，说明仅几行，Token 消耗几乎可以忽略。

### 安全性的真实差距

CLI 的风险：命令无边界，AI 生成的命令如果夹带 `rm -rf`，后果不可逆。文件名包含特殊字符时，命令容易出错且难以发现。

MCP 的优势：JSON 传参，文件名老老实实躺在一个字段里，无论包含什么特殊字符都能正确处理。且只能执行设计者允许的操作，天然有边界。

### 该怎么选

| 场景 | 推荐 | 原因 |
|------|------|------|
| 本地开发、个人脚本 | CLI | 快、省 Token、灵活 |
| 企业级自动化流程 | MCP | 安全、可控、可审计 |
| 云端/多用户环境 | MCP | 隔离性好 |
| 常见工具（git、grep 等） | CLI | AI 训练时大量见过，直接能用 |
| 冷门/自定义工具 | MCP | 结构化描述让 AI 更容易正确调用 |

**结论**：MCP 解决接入问题，CLI 解决执行问题。它们不是替代关系，而是在 Agent 时代被重新分工了。CLI 的使用比例会越来越大，但 MCP 不会消失——只要存在大量异构系统和工具，统一接入的问题就永远存在。
