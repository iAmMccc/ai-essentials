# MCP 与 CLI 技术理解

## 一、MCP 是什么

MCP（Model Context Protocol，模型上下文协议）是 Anthropic 于 2024 年 11 月发布的开放标准协议。

**一句话理解**：AI 大模型本质上是一个封闭系统——你给它文本，它返回文本。它看不到你的文件、连不上你的数据库、调不了你的 API。MCP 就是给 AI 装上"手脚"的标准接口，让它能触达真实世界。

**类比**：USB 出现之前，每种外设都有自己的接口；USB 出现之后，一个标准走天下。MCP 就是 AI 世界的 USB。

---

## 二、MCP 解决什么问题

### 没有 MCP 之前

每对接一个外部工具，就要写一套定制代码：

```
接 GitHub → 写一套
接 Slack  → 再写一套
接数据库  → 再写一套
换一个 AI 工具 → 全部重写
```

M 个 AI 应用 × N 个外部工具 = **M × N 个适配器**，工作量随工具数量爆炸增长。

### 有了 MCP 之后

工具端实现一次 MCP Server，就能被所有支持 MCP 的 AI 调用；AI 端实现一次 MCP Client，就能接入所有 MCP Server。**M × N 变成了 M + N**。

---

## 三、MCP 的架构

### 三个角色

| 角色 | 作用 | 示例 |
|------|------|------|
| **Host（宿主）** | 面向用户的 AI 应用 | Claude Desktop、Cursor、Claude Code |
| **Client（客户端）** | Host 内部负责通信的模块 | 自动管理，用户无需关心 |
| **Server（服务端）** | 工具和数据源的提供方 | GitHub Server、数据库 Server、Figma Server |

### 一次调用的流程

以"查询最近 7 天活跃用户数"为例：

```
你说："查一下最近 7 天的活跃用户数"
    ↓
AI 理解意图，决定需要查询数据库
    ↓
AI 通过 MCP 协议调用数据库 Server
    ↓
Server 执行 SQL 查询，返回结果
    ↓
AI 基于结果生成回答："最近 7 天活跃用户 12,483 人，环比增长 8.2%"
```

### 三大核心能力

| 能力 | 说明 | 控制方 | 示例 |
|------|------|--------|------|
| **Resources（资源）** | 只读数据源 | 应用程序 | 文件内容、数据库记录、API 响应 |
| **Tools（工具）** | 可执行的函数 | AI 模型 | 搜索航班、发送消息、创建日历事件 |
| **Prompts（模板）** | 预制的 Prompt 模板 | 用户 | 计划旅行、总结会议、起草邮件 |

最常用的是 **Tools**——AI 判断需要执行某个操作时，调用对应的 Tool，拿到结果后继续推理。

### 两种通信方式

| 模式 | 通信方式 | 适用场景 |
|------|---------|---------|
| **Stdio** | 本地进程间通信（stdin/stdout） | 本地工具：文件操作、命令执行 |
| **HTTP** | 网络请求 | 远程服务：云端 API、跨机器调用 |

## 四、MCP 的安全机制

AI 能调用外部工具了，怎么保证它不乱来？

| 机制 | 说明 |
|------|------|
| **权限控制** | 可限制 Server 只暴露特定 Tools（如只允许读，不允许写） |
| **操作确认** | AI 调用工具前可要求用户确认，而非静默执行 |
| **操作审计** | 所有工具调用都有记录，可追溯谁在什么时候调了什么 |
| **沙箱隔离** | MCP Server 作为独立进程运行，与 AI 应用隔离 |

## 五、CLI 是什么

CLI（Command Line Interface）就是在终端里执行的命令行工具，如 `git`、`grep`、`gh` 等。

在 AI 编程场景中，CLI 模式指的是：**AI 不通过 MCP 调用专用工具，而是直接生成一条 shell 命令，通过 `bash` 执行**。

---

## 六、MCP vs CLI 对比

### 执行流程对比

**MCP 模式**（以批量处理照片为例）：

```
AI 思考 → 调用"读取目录"工具        → 结果返回 AI
AI 思考 → 调用"读取图片信息"工具     → 结果返回 AI
AI 思考 → 调用"图像处理/加水印"工具  → 结果返回 AI
AI 思考 → 调用"上传"工具           → 结果返回 AI
AI 思考 → 输出最终答案
```

每一步都要经过 AI 调度，**AI 是整个链路的调度中心**。

**CLI 模式**（同样的任务）：

```
AI 思考 → 生成一条组合命令 → 调用 bash 执行
命令在本地自动跑完全部操作（无需 AI 介入）
结果返回 AI → 输出最终答案
```

示例命令：
```bash
exiv2 [筛选横版照片] | convert [ImageMagick 加水印，输出到 output/] && scp [上传到服务器]
```

### 核心差异

| 维度 | CLI | MCP |
|------|-----|-----|
| **Token 消耗** | 极低（只传一个 bash 命令） | 高（所有工具描述都要传入上下文） |
| **执行效率** | 高（一次调用完成整个流程） | 低（多轮调度，每步都经过 AI） |
| **灵活性** | 最高（管道符自由组合） | 低（需求变了要重新开发工具） |
| **可控性** | 低（特殊字符易出错） | 高（JSON 结构化，参数与命令分离） |
| **安全性** | 低（命令无边界，可能误删文件） | 高（操作受限，只能执行设计者允许的操作） |
| **适用场景** | 本地开发、个人使用 | 企业级、云端、高安全要求 |

### Token 消耗差异详解

以 GitHub MCP Server 为例：
- 共 44 个工具
- 全部工具说明约 163,700 字符
- 换算约 **14,268 个 Token**
- 按 Claude Sonnet 定价约 **3 美分/次**
- 如果同时启用多个 MCP Server，仅工具说明就可能花费数美分

而 CLI 方式只需传入一个 `bash` 工具，说明仅几行，Token 消耗几乎可以忽略。

### 安全性差异详解

**CLI 的风险**：命令无边界，AI 生成的命令如果夹带 `rm -rf` 类操作，本地文件可能被误删。文件名包含特殊字符（如引号、空格）时，命令容易出错且难以发现。

**MCP 的优势**：工具使用 JSON 传参，文件名老老实实躺在一个字段里，无论包含什么特殊字符都能正确处理。且只能执行工具设计者允许的操作，天然有边界。

## 七、该用 MCP 还是 CLI

**不是二选一，而是各取所长**：

| 场景 | 推荐 | 原因 |
|------|------|------|
| 本地开发、个人脚本 | CLI | 快、省 Token、灵活 |
| 企业级自动化流程 | MCP | 安全、可控、可审计 |
| 云端/多用户环境 | MCP | 隔离性好，一条失控命令不会影响其他用户 |
| 常见工具（git、grep 等） | CLI | AI 训练时已大量见过，直接能用 |
| 冷门/自定义工具 | MCP | 提供结构化工具描述，AI 更容易正确调用 |
| 对安全性要求高 | MCP | 操作受限、可审计 |
| 追求极致效率 | CLI | 一条命令跑完整个流程 |

**趋势判断**：
- CLI 的使用比例会越来越大——大多数场景下更快、更便宜
- MCP 不会消失——会聚焦在企业级、云端等高安全要求的场景
- 两者最终会共存，服务于不同的应用场景

## 八、常见 MCP Server

社区已经有大量现成的 MCP Server：

| MCP Server | 能力 | 典型场景 |
|-----------|------|---------|
| **figma** | 读取设计稿结构化数据 | 设计转代码 |
| **filesystem** | 读写本地文件 | 读取配置、生成代码文件 |
| **github** | 操作 GitHub 仓库 | 创建 PR、查看 Issue |
| **postgres / mysql** | 数据库查询 | 数据分析 |
| **browser** | 浏览器自动化 | 网页抓取、前端测试 |
| **sentry** | 错误监控 | 崩溃分析、日志查询 |
| **slack / 飞书** | 消息通知 | 发送通知、查询聊天记录 |

不需要自己写这些 Server，直接配置即可使用。

查找更多：[MCP 官方 Server 仓库](https://github.com/modelcontextprotocol/servers)

## 九、MCP 与相关概念的区别

| 概念 | 层次 | 作用 | 类比 |
|------|------|------|------|
| **Function Calling** | 调用机制 | LLM 发出"我要调用某个函数"的指令 | 拍一下手（发出指令） |
| **Plugin** | 工具封装 | 把某个工具的 API 包装成 AI 可调用的格式 | 一条手臂上的关节 |
| **MCP** | 标准协议 | 定义 AI 和所有工具之间的通信标准 | USB 接口标准 |

关键区别：**Function Calling 是特定厂商的 API 特性**（如 OpenAI 的 Function Calling 只能用于 OpenAI 模型），**MCP 是开放标准**，任何 AI 模型都可以通过它接入工具。
