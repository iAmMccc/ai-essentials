# Agent

> 理解 AI

Agent（智能体）让 AI 从「只会说」变成「能动手」— 它不只是回答你的问题，而是能自主规划步骤、调用工具、根据结果调整行动，直到任务完成。

你用 Claude Code 让它「帮我重构这个模块」，它会自己读代码、分析问题、改文件、跑测试、修 Bug — 这背后就是 Agent 在工作。它不是一问一答，而是一个持续运转的「思考 → 行动 → 观察 → 再思考」的循环。

理解 Agent 是怎么工作的，你才能知道：为什么有时候它能跨几十个文件完成重构，有时候却在简单任务上绕圈子；为什么拆任务比塞一个大 Prompt 效果好；为什么子 Agent 并行能提升效率。

---

## 一、Agent 和普通 AI 对话的区别

普通的 AI 对话是单轮的：你问一个问题，AI 生成一段文字，结束。AI 没有「手」，它不能读文件、不能执行命令、不能调用 API — 只能用文字告诉你该怎么做，然后你自己去做。

Agent 在 AI 的「大脑」之上，加了四样东西：

| 组件 | 作用 | 类比 |
|------|------|------|
| **工具（Tools）** | 让 AI 能执行操作：读写文件、跑命令、调 API | 手和脚 |
| **规划（Planning）** | 把复杂任务拆成多个步骤 | 项目管理能力 |
| **记忆（Memory）** | 记住之前做了什么、结果是什么 | 工作笔记 |
| **循环（Loop）** | 反复「思考 → 行动 → 观察结果 → 再思考」直到完成 | 持续工作的能力 |

一句话概括：**Agent = LLM + 工具 + 规划 + 记忆 + 执行循环。**

用 iOS 开发的视角来理解：普通 AI 对话像是一个只能给你口头建议的顾问，Agent 像是一个能打开 Xcode、读代码、改文件、跑测试的开发者。区别不在于它多聪明，而在于它能不能动手。

## 二、Agent 怎么工作：两种执行模式

Agent 拿到一个任务后，怎么决定先做什么、再做什么？主流有两种模式。

### 模式一：ReAct（边想边做）

ReAct（Reasoning + Acting）是目前使用最广泛的模式。核心逻辑是交替进行「思考」和「行动」：

```
Thought（思考）：分析当前情况，决定下一步
    ↓
Action（行动）：调用一个工具
    ↓
Observation（观察）：拿到工具返回的结果
    ↓
Thought（思考）：根据结果决定继续做什么
    ↓
... 循环 ...
    ↓
Final Answer（完成）
```

一个实际的例子 — 你让 Agent「帮我查一下 NetworkService 里有没有内存泄漏」：

> **Thought**：需要先读取 NetworkService 的代码。
>
> **Action**：读取 `Services/NetworkService.swift`
>
> **Observation**：（文件内容返回）
>
> **Thought**：看到第 42 行的闭包捕获了 self 但没有用 weak。需要检查这个闭包是否会被长期持有。
>
> **Action**：搜索这个闭包被赋值给了什么属性
>
> **Observation**：闭包被赋值给了 `completionHandler`，是实例属性，会形成循环引用。
>
> **Final Answer**：NetworkService.swift 第 42 行存在循环引用风险...

每一步都是「想一下 → 做一下 → 看结果 → 再想」，像一个有经验的开发者在排查问题。

**ReAct 的特点**：灵活，能根据中间结果随时调整方向。但因为每步都要调用一次 LLM，Token 消耗较高，复杂任务可能绕路。

### 模式二：先规划再执行（Plan-then-Execute）

另一种模式是先想好所有步骤，再按计划逐步执行：

```
第一阶段：规划
    用户目标 → LLM 生成完整的步骤列表（TODO）

第二阶段：执行
    逐步执行每个步骤 → 每步可能调用工具 → 必要时修正计划
```

Claude Code 里经常能看到这种模式 — 你让它实现一个功能，它先输出「我将做 1、2、3...」的计划，然后逐项执行。

**Plan-then-Execute 的特点**：整体感强，不容易在细节里迷路。但前期规划依赖对任务的准确理解，如果理解偏了，整个计划都要推倒重来。

### 两种模式的对比

| | ReAct | Plan-then-Execute |
|---|---|---|
| **决策时机** | 每步都重新决策 | 先做完整规划，再按计划走 |
| **灵活性** | 高，随时调整 | 中，修改计划成本较高 |
| **适合场景** | 探索性任务、调试 | 步骤明确的实现任务 |
| **Token 消耗** | 较高（每步都推理） | 较低（规划一次，执行按步走） |

实际产品中两者经常结合：先做一层高层规划（TODO 列表），每个 TODO 内部用 ReAct 来执行。Claude Code 就是这么干的。

## 三、子 Agent：拆分任务的关键手段

一个 Agent 面对复杂任务时，不一定要自己全干完。它可以创建**子 Agent（Subagent）**— 每个子 Agent 是一个独立的工作单元，有自己的上下文窗口、工具集和对话历史。

### 为什么需要子 Agent

两个核心原因：

**上下文隔离** — 主 Agent 在长对话中会积累大量历史信息，上下文窗口逐渐被填满，输出质量下降。子 Agent 从干净的上下文开始工作，不会被主 Agent 之前的对话「污染」。

**并行执行** — 多个子 Agent 可以同时工作。比如你要重构三个模块，可以同时派三个子 Agent 分别处理，而不是线性地一个一个来。

### 子 Agent 的工作方式

主 Agent 创建子 Agent 时，会传给它一段 Prompt 描述任务。这段 Prompt 是子 Agent 的**唯一上下文** — 主 Agent 之前读过的文件、聊过的内容，子 Agent 看不到。

```
主 Agent
    ├── 子 Agent 1：「重构 LoginService，改用 async/await」
    ├── 子 Agent 2：「给 PaymentService 加单元测试」
    └── 子 Agent 3：「更新 API 文档」
```

每个子 Agent 独立完成任务后，把结果返回给主 Agent。主 Agent 汇总结果，继续推进。

这也解释了一个常见问题：为什么有时候子 Agent 的结果不够好？因为主 Agent 传给它的 Prompt 可能缺少关键背景。你可以在对话中显式补充信息，让主 Agent 写出更完整的子任务描述。

### Claude Code 中的三种内置 Agent

| Agent | 工具权限 | 用途 |
|-------|---------|------|
| **Explore** | 只读（读文件、搜索） | 搜代码、找文件、理解代码库结构 |
| **Plan** | 只读 | 架构设计、实现方案规划 |
| **General Purpose** | 全部 | 复杂的多步骤任务 |

Explore 最轻量 — 只读模式、不加载项目规则文件、启动快。适合「帮我找一下这个函数在哪里被调用」这类搜索任务。

你也可以创建自定义 Agent。在 `.claude/agents/` 下建一个 `.md` 文件：

```yaml
---
name: code-reviewer
description: 审查代码变更，关注安全和性能
tools: Read, Grep, Glob
model: sonnet
---

你是一个代码审查专家。收到审查请求后：
1. 用 git diff 查看所有变更
2. 逐文件检查安全漏洞、性能问题、错误处理
3. 按严重程度分级输出审查意见
```

`description` 字段不只是文档 — AI 会根据它自动判断什么时候该用这个 Agent。写得模糊就不会被自动调用。

## 四、Agent 的能力边界

Agent 很强，但不是万能的。理解它的边界，才能用对场景。

### Agent 做得好的事

- **跨多文件的代码修改**：能读懂项目结构，在正确的位置做正确的修改
- **探索性任务**：不确定答案在哪，需要搜索、阅读、分析、再搜索
- **重复性的多步骤操作**：批量修改、批量测试、批量生成
- **需要工具配合的任务**：查日志 + 读代码 + 改代码 + 跑测试的组合

### Agent 做不好的事

- **需要创造性突破的设计**：Agent 擅长在已知模式上组合，不擅长从零创新
- **需要深层业务理解的决策**：它能读你的代码，但不理解你的商业逻辑
- **对精确性要求极高的场景**：Agent 的每一步决策都有概率出错，步骤越多，累积误差越大

### 使用建议

**长任务拆成阶段** — 不要在一个 Agent 里干到上下文满。研究阶段用 Explore，规划阶段用 Plan，实现阶段用 General Purpose。每个阶段是独立的上下文，不会互相污染。

**并行而非串行** — 多个独立的子任务，派多个子 Agent 同时做。5 个并行 Agent 的总工作空间远大于 1 个 Agent 线性处理。

**独立 Agent 做 Review** — 实现完功能后，启动一个新的子 Agent 做代码审查。为什么不用主 Agent 自己 Review？因为主 Agent 在实现过程中形成了「这样是对的」的上下文偏见，独立子 Agent 没有这个包袱，审查更客观。

**给足背景信息** — 子 Agent 从零开始工作，主 Agent 之前的对话它看不到。如果你需要子 Agent 了解项目背景，在对话中显式说出来：「我们用的是 JWT + Redis session，入口在 src/auth/middleware.ts」，这样主 Agent 传给子 Agent 的 Prompt 会包含这些关键信息。

## 五、Agent 与其他概念的关系

Agent 是这个系列里出现过的所有概念的「使用者」：

| 概念 | Agent 怎么用它 |
|------|--------------|
| [Prompt](../../using/prompt/) | Agent 的每一步推理都是一次 Prompt 调用 |
| [Rules](../../using/rules/) | Agent 在执行过程中自动遵守项目规范 |
| [Skills](../../using/skills/) | Agent 匹配到相关 Skill 后按标准流程执行 |
| [Projects](../../using/projects/) | Agent 在项目上下文中工作，读取代码和文档 |
| [MCP](../../using/mcp/) | Agent 通过 MCP 调用外部工具和数据源 |

它们的协作关系：

```
用户下达任务
    ↓
Agent 理解任务，开始规划
    ↓
按 Skill 的标准流程执行（如果匹配到）
    ↓
执行过程中遵守 Rules
    ↓
在 Projects 提供的上下文中工作
    ↓
需要外部工具时通过 MCP 调用
    ↓
必要时创建子 Agent 并行处理子任务
    ↓
汇总结果，完成任务
```

如果前面几篇讲的是 AI 的「知识」「规范」「技能」「记忆」「手脚」，那 Agent 就是把这些全部串起来的「人」— 它决定什么时候用什么能力，按什么顺序推进，直到任务完成。

## 六、小结

| 维度 | 要点 |
|------|------|
| **Agent 是什么** | LLM + 工具 + 规划 + 记忆 + 执行循环，能自主完成多步骤任务 |
| **和普通对话的区别** | 普通对话只能「说」，Agent 能「做」— 读文件、改代码、跑测试 |
| **两种执行模式** | ReAct（边想边做，灵活）和 Plan-then-Execute（先规划再执行，有全局感） |
| **子 Agent** | 独立的工作单元，有自己的上下文，可并行执行 |
| **能力边界** | 擅长跨文件修改、探索、重复操作；不擅长创造性设计和深层业务判断 |
| **核心使用建议** | 长任务拆阶段、独立子任务并行、新 Agent 做 Review、给子 Agent 足够背景 |
