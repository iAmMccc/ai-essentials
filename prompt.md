# Prompt

> AI 工作体系入门

Prompt（提示词）是你与 AI 沟通的语言，也是决定 AI 输出质量的第一道关卡。

同样一个 AI 模型，有人用它生成的代码可以直接跑通，有人却连编译都过不了——差距往往不在模型本身，而在你怎么跟它说话。

本文从概念、作用、底层原理、写法技巧、调试方法、实战示例、常用资源和局限性八个方面，帮你建立一套系统的 Prompt 认知与实操方法。读完之后，你不仅知道 Prompt 是什么，更能写出让 AI 稳定输出高质量结果的 Prompt。

---

## 一、Prompt 是什么

### 1. 定义

Prompt（提示词）是用户给 AI 的输入指令。它可以是一句话、一段任务描述，也可以是一个结构化模板。

比如下面这些，都是 Prompt：

- 「帮我写一段 Python 代码，实现用户登录验证」
- 「用小学生能理解的话解释什么是 HTTP」
- 「从产品经理视角，分析一下小红书和抖音的差异」

简单说，**你对 AI 说的每一句话，都是 Prompt**。

### 2. 核心特点

- **即时性**：Prompt 只存在于当前对话中。如果不保存，下次对话 AI 不会记得你上次说了什么。
- **灵活性**：你可以随时调整措辞、补充条件、换个角度重新提问，来改变 AI 的输出。
- **门槛低，上限高**：任何人都能写 Prompt，但写出能稳定产出高质量结果的 Prompt，需要刻意练习。

### 3. Prompt 与相关概念的区别

在 AI 工具的体系里，除了 Prompt，你还会遇到 Rules、Skills 等概念。它们各司其职：

| 概念 | 作用 | 特点 |
|------|------|------|
| **Prompt** | 告诉 AI「这一次做什么」 | 一次性指令，用完即消 |
| **Rules** | 告诉 AI「什么该做、什么不该做」 | 持久化的行为规范，自动生效 |
| **Skills** | 告诉 AI「这类事按什么流程做」 | 可复用的技能包，包含指令 + 工具 + 流程 |

打个比方：Prompt 像是你在餐厅点菜时说的那句话，Rules 是厨房的卫生规范，Skills 是大厨的拿手菜谱。三者配合，才能端出一桌好菜。

> 本文聚焦 Prompt。Rules、Skills 等概念会在后续文章中详细展开。

## 二、Prompt 为什么重要

### 1. 同样的 AI，Prompt 不同，结果天差地别

先看一个对比：

| | 模糊 Prompt | 清晰 Prompt |
|---|---|---|
| **输入** | 「写个登录功能」 | 「用 Swift + UIKit 实现登录页，包含账号密码输入框、表单校验、网络请求，使用 async/await，不用第三方库」 |
| **AI 表现** | 可能给你一段没有校验、没有错误处理、甚至语言都不对的代码 | 直接输出一份结构清晰、可以跑通的登录模块 |

输入越精确，AI 的输出质量就越高。这不是玄学，而是由大模型的工作原理决定的（第三节会详细解释）。

### 2. Prompt 的三个核心作用

一个好的 Prompt 同时在做三件事：

**控制方向** — 告诉 AI 要做什么、从什么角度做。

> 「解释 TCP」 → AI 可能从任意角度泛泛而谈。
>
> 「从 iOS 开发者的角度，解释 TCP 三次握手在网络请求中的实际影响」 → AI 会聚焦到你真正关心的层面。

**提供上下文** — 给 AI 足够的背景信息，减少它的猜测空间。

> 你是一名资深系统架构师，请设计一个支持千万级用户访问的订单系统，需要包含架构图、核心模块和数据库设计。

这一句话同时给了角色（架构师）、任务（设计订单系统）、约束（千万级）、输出形式（架构图 + 模块 + 数据库设计）。AI 收到的信息越完整，输出就越贴合你的预期。

**施加约束** — 限定输出的范围、格式和风格，防止 AI 自由发挥跑偏。

> - 「用 200 字以内解释什么是 GitHub」
> - 「请用表格形式输出」
> - 「只列要点，不要展开论述」

一句话总结：**好的 Prompt = 方向明确 + 上下文充分 + 约束具体**。

### 3. 学习 Prompt 的深层价值

学习写好 Prompt，收益不仅仅是「AI 听话了」：

- **提升表达精度**：写 Prompt 的过程，本质上是在梳理「我到底要什么」。很多时候你觉得 AI 输出不好，回头看看 Prompt，往往是自己没想清楚。
- **节省时间成本**：一个精准的 Prompt 可以一次到位，省去反复沟通、多轮修改的时间。
- **建立结构化思维**：Prompt 的写法（角色、背景、任务、约束、输出）其实就是一套需求分析框架，这个能力在任何协作场景中都有用。

## 三、Prompt 的底层原理

很多人写不好 Prompt，不是因为技巧不够，而是不理解大模型到底在做什么。搞懂底层原理，你就能从「凭感觉写」变成「有针对性地设计」。

### 1. 大模型是一台「概率接龙机」

大模型（如 GPT、Claude、DeepSeek）的核心工作方式可以简化为三步：

```
你输入的 Prompt（上下文）
        ↓
模型计算每个 Token 出现的概率
        ↓
选出概率最高的 Token，拼接成回答
```

也就是说，**模型不是在「思考」你的问题，而是在做超大规模的文本接龙** — 根据你给的前文，预测最可能出现的下文。

这有点像手机输入法的联想功能：你打「今天天气」，输入法会建议「不错」「很好」，因为这些词在训练数据中出现频率最高。大模型做的事情本质上一样，只不过它的「联想」范围覆盖了整个人类知识库，精度也高出几个数量级。

### 2. 理解原理后的四个关键认知

**(1) AI 不理解世界，它理解的是文本模式**

当你问「解释 TCP 三次握手」，模型并不是在思考网络协议的工作机制。它做的是：在海量训练数据中，找到「解释 TCP 三次握手」这个上下文最常出现的后续文本，然后生成一段最像「正确解释」的回答。

它靠的是语言中的统计规律，而不是对协议本身的理解。这就是为什么 AI 有时候能写出看似正确但细节有误的内容 — 因为它追求的是「像」，不是「对」。

**(2) Prompt 越清晰，模型「猜」得越少**

因为模型是在做概率预测，所以：

- **模糊的 Prompt** → 概率分布分散 → 模型靠「猜」来补全 → 结果不可控
- **清晰的 Prompt** → 概率分布集中 → 模型生成范围被锁定 → 结果更稳定

比如：「写段代码」→ 模型要猜语言、猜功能、猜风格，每一步都可能偏。「用 Python 写一个快速排序函数，要求包含类型注解和详细注释」→ 所有关键变量都被锁定，模型几乎没有猜测空间。

**(3) Prompt 不是「提问」，而是「上下文编程」**

这是理解 Prompt 最重要的思维转换。

很多人把 Prompt 当作「提一个问题，等 AI 回答」。但更准确的理解是：**你在用自然语言给模型写一段临时程序，配置它的输出行为**。

比如这段 Prompt：

> 你是一名资深 iOS 工程师。请分析下面代码的线程安全问题。只输出风险点。

拆开看，就是在做「编程」：

| Prompt 中的文字 | 实际作用 |
|---|---|
| 你是一名资深 iOS 工程师 | 设定角色 = 配置模型的知识域 |
| 请分析下面代码的线程安全问题 | 定义任务 = 配置模型的执行目标 |
| 只输出风险点 | 设置约束 = 配置输出过滤条件 |

理解这一点之后，你写 Prompt 时就不会再「随口一问」，而是会有意识地去「设计」每个部分。

**(4) 上下文有容量限制**

大模型有一个「上下文窗口」（Context Window），即它一次能处理的最大文本量，目前主流模型在 128K ~ 200K Token 左右。超过这个限制，最早输入的内容会被截断或压缩。

这意味着：你不能把整个项目的代码和文档全部塞给 AI。它始终是在有限的信息下工作，所以**提供高质量的、精炼的上下文，比提供大量低质量的信息更重要**。

## 四、如何写好 Prompt

理解了原理之后，接下来是实操。这一节从「写什么」「怎么写」「怎么避坑」三个层面，给你一套可落地的方法。

### 1. 好 Prompt 的核心要素

写 Prompt 之前，先过一遍这五类信息。不需要每次都写全，但心里要有这个框架：

| 要素 | 作用 | 示例 |
|------|------|------|
| **角色** | 让 AI 站在特定专业视角 | 「你是一名有 10 年经验的后端架构师」 |
| **任务** | 明确要做什么 | 「设计一个用户权限系统」 |
| **背景** | 补充项目、场景、技术栈等上下文 | 「项目使用 Spring Boot 3 + MySQL」 |
| **输入** | 提供需要处理的素材 | 「以下是现有的数据库表结构：...」 |
| **输出** | 定义期望的格式、长度、风格 | 「用 Markdown 表格输出，不超过 500 字」 |

**最小可用组合**：任务 + 输出格式。即使只写这两条，效果也比「裸问」好很多。

**完整示例**：

> 你是一名资深 iOS 工程师（角色），
> 请分析下面 Swift 代码在网络请求场景下可能存在的问题（任务 + 背景），
> 重点关注线程安全和循环引用（约束），
> 只列出风险点，不需要解释基础语法（输出格式）。

### 2. 常用结构模板

不同场景适合不同的 Prompt 结构。下面是四个最实用的模板，从简到繁排列。

#### (1) 角色 + 任务 + 格式

最简模板，适合快速提问和简单任务。

```
你是 [角色]。
请 [任务]。
输出要求：[格式/长度/形式]。
```

**示例**：

> 你是一名资深 iOS 工程师。
> 请解释 Swift 中 escaping closure 的使用场景。
> 输出要求：结合网络请求回调举例，并给出代码示例。

#### (2) 背景 + 任务 + 约束

适合工程问题、项目相关的任务，需要给 AI 交代技术背景。

```
背景：[项目/技术栈/当前状况]
任务：[要完成的事]
约束：[必须做什么/禁止什么/偏好什么]
```

**示例**：

> **背景**：我们正在开发一个 iOS App，需要实现图片列表加载。
>
> **任务**：设计一个图片加载方案。
>
> **约束**：
> - 使用 Swift
> - 不使用第三方库
> - 需要支持内存 + 磁盘双级缓存
> - 列表滚动时避免重复下载

#### (3) Few-shot：给示例引导

当你需要 AI 严格按照特定格式输出时，最有效的方法不是描述格式，而是直接给例子。

**示例**：

```
请把用户反馈分类为：功能建议 / Bug 报告 / 其他

示例：
输入：希望支持深色模式
输出：功能建议

输入：点击保存会闪退
输出：Bug 报告

现在请分类：
输入：登录时一直转圈
```

给了示例之后，模型会非常稳定地模仿这个格式输出。这个技巧在批量处理、数据标注类任务中尤其好用。

#### (4) CRISPE 框架（复杂任务）

当任务比较复杂，涉及多个维度时，可以用 CRISPE 框架来组织信息：

| 字母 | 含义 | 作用 |
|------|------|------|
| **C** | Capacity / Role | 定义角色和能力范围 |
| **R** | Request | 明确任务请求 |
| **I** | Input | 提供输入信息 |
| **S** | Steps | 指定执行步骤 |
| **P** | Personality | 设定回答风格 |
| **E** | Experiment | 要求多个方案对比 |

**示例**：

> **Role**：资深 iOS 架构师
>
> **Request**：根据 PRD 设计客户端技术方案
>
> **Input**：以下是产品需求文档摘要：（粘贴内容）
>
> **Steps**：
> 1. 分析核心业务模块
> 2. 设计数据模型
> 3. 给出接口结构和数据流
>
> **Personality**：回答风格偏工程化，直接给方案，不要过多理论铺垫

实际使用中不需要六个元素全写，根据任务复杂度灵活选用。

> 除了 CRISPE，社区中还有 BROKE（Background-Role-Objective-Key constraints-Examples）、ICIO（Instruction-Context-Input-Output）等框架，核心思路都类似：结构化地组织角色、任务、背景、约束和输出要求。不必纠结用哪个框架，找到一个适合自己的记住就好。

### 3. 进阶技巧

掌握了基础模板之后，以下几个技巧能进一步提升 Prompt 的效果：

#### (1) 思维链：让 AI「一步步想」

对于复杂的推理、分析或 Debug 问题，在 Prompt 中加一句「请一步步分析」（Think step by step），可以显著提高准确率。这个技巧被称为**思维链（Chain of Thought, CoT）**，相当于让 AI 在回答之前先「打草稿」。

**示例**：

> 我遇到了一个 `ConcurrentModificationException`。下面是代码：（粘贴代码）
>
> 请一步步分析：在这个 ArrayList 遍历过程中，哪个操作导致了并发修改，为什么会触发异常，以及如何修复。

不加思维链时，AI 可能直接给一个笼统的建议。加了之后，它会按步骤展开推理过程，答案的准确性和可验证性都会提升。

#### (2) 分隔符：把指令和数据分开

当你的 Prompt 同时包含指令和输入数据（如代码片段、日志、文档）时，用分隔符明确区分两者，防止模型把数据当成指令来执行。

常用分隔符：`"""`、`---`、`###`、XML 标签（如 `<code>...</code>`）。

**示例**：

```
请解释下面 """ 包裹的代码中的内存泄漏风险：

"""
public class Cache {
    private static final Map<String, Object> map = new HashMap<>();
    // ...
}
"""
```

这个习惯在处理长文本、多段输入时尤其重要，能有效减少 AI 的「理解偏差」。

#### (3) 结构化 Prompt：像写代码一样写指令

当 Prompt 变得复杂时，可以借鉴编程的思路，用 Markdown 结构来组织。社区中的 [LangGPT](https://github.com/langgptai/LangGPT) 框架就是这个思路的典型代表，它用 `Role`、`Skills`、`Rules`、`Workflow` 等模块来构建 Prompt：

```markdown
# Role: iOS 代码审查专家

## Profile
- Version: 1.0
- Language: 中文
- Description: 专注于 Swift/iOS 代码质量审查

## Skills
1. 识别内存泄漏、循环引用等常见问题
2. 评估代码的可读性和可维护性

## Rules
1. 只指出实际存在的问题，不要过度解读
2. 每个问题附上具体的代码行号和修改建议

## Workflow
1. 先通读代码，理解整体结构
2. 按优先级列出问题（严重 → 一般 → 建议）
3. 给出修改后的代码示例
```

这种结构化写法的好处是：模块清晰、易于复用、方便迭代。当你的某个 Prompt 需要反复使用时，把它结构化是一个值得投入的优化。

### 4. 三个设计原则

技巧之上，还需要几个原则来指导日常写作：

**原则一：精准描述，减少歧义**

AI 不会「猜你的意思」，它只会按字面理解。

| 写法 | 问题 |
|------|------|
| ❌ 「优化一下这段代码」 | AI 不知道优化什么：性能？可读性？安全性？ |
| ✅ 「重构这段代码，将嵌套回调改为 async/await，保持原有功能不变」 | 目标明确，AI 能精准执行 |

**原则二：拆分大任务，逐步推进**

复杂需求不要一股脑塞进一个 Prompt，拆成多步效果更好：

> **第一步**：「分析这段代码的架构问题，列出 3 个最严重的问题」
>
> **第二步**：「针对第 1 个问题，给出重构方案和代码」
>
> **第三步**：「检查重构后的代码是否有性能问题」

每次只做一件事，AI 的输出质量会显著提升。

**原则三：明确边界，控制发挥空间**

用「只…」「不要…」「必须包含…」来划定边界：

> - 「只列出风险点，不需要解释基础概念」
> - 「不要使用第三方库」
> - 「必须包含错误处理逻辑」

边界越清晰，AI 越不容易跑偏。

### 5. 常见问题与解决方式

| 问题 | 常见原因 | 解决方式 |
|------|---------|---------|
| 输出太泛，不够深入 | 没有限定角色、场景和受众 | 加上角色设定和具体场景 |
| 输出太长，废话多 | 没有格式和长度约束 | 加「200 字以内」「只列要点」 |
| 答非所问，跑题 | 一个 Prompt 混了多个任务 | 拆分成多个 Prompt，每次只做一件事 |
| 格式不符合预期 | 没有给输出示例 | 用 Few-shot 给一个格式范例 |
| 内容重复或自相矛盾 | 上下文太长，模型丢失焦点 | 精简上下文，只保留关键信息 |
| 代码无法运行 | 没有指定技术栈版本和约束 | 写清语言、框架、版本（如「Swift 5.9 + iOS 17」） |

### 6. Prompt 的工程化管理

当你写的 Prompt 越来越多，值得考虑系统化管理：

**模板化** — 把常用 Prompt 固定为模板，每次只替换变量部分：

```
角色：[         ]
任务：[         ]
背景：[         ]
输入：[         ]
输出格式：[     ]
约束：[         ]
```

**版本管理** — Prompt 改一个词，输出可能完全不同。给你的 Prompt 编版本号（v1、v2、v3），便于对比效果和回滚。

**团队沉淀** — 在团队内部建立 Prompt Library，按场景分类（代码生成、代码审查、文档写作、Bug 分析……），避免每个人重复造轮子。

## 五、Prompt 调试方法

好的 Prompt 不是一次写出来的，而是调出来的。下面是一个实战中非常有效的三步调试法。

### 第一步：写最简版本，建立基线

先用最简单的方式表达你的需求：

> 解释 Swift 中的 escaping closure

看 AI 怎么回答。这一步的目的不是拿到满意结果，而是建立一个「基线」— 知道模型在没有额外约束时会怎么回答。

### 第二步：对照输出，诊断问题

拿到基线输出后，逐项检查：

| 问题类型 | 表现 |
|---------|------|
| 太浅 | 只讲了概念定义，没有深入到实际场景 |
| 太泛 | 什么都提了一嘴，但没有针对你的需求 |
| 缺示例 | 全是文字描述，没有代码 |
| 偏离场景 | 讲的是通用场景，和你的项目上下文不匹配 |

标出具体是哪一类问题（通常不止一个），然后针对性地补充。

### 第三步：补约束，迭代优化

根据诊断出的问题，在原有 Prompt 上叠加约束：

> 从 iOS 开发角度解释 Swift 中的 escaping closure，结合网络请求回调场景，重点说明循环引用问题及其解决方案，并给出完整的代码示例。

对比第一步的输出，检查改进是否到位。如果还不满意，继续微调，直到输出稳定达标。

### 经验总结

**Prompt = 写一半 + 调一半**。

不要追求一次写出完美的 Prompt。先写一个「能跑」的版本，看看输出哪里不行，然后有针对性地加约束、加示例、加角色。这种迭代方式，比花 10 分钟冥思苦想一个「完美 Prompt」要高效得多。

## 八、Prompt 的局限

Prompt 虽然是与 AI 交互的基础，但它有明确的能力边界。了解这些局限，才能知道什么时候该用 Prompt，什么时候该升级到更高级的工具。

### 1. 上下文窗口有限

大模型一次能处理的文本量（Context Window）是有上限的，目前主流模型在 128K ~ 200K Token 之间。你无法把整个代码仓库和全部文档一次性塞给 AI，它始终是在局部信息下工作。信息不全时，AI 会用自己的「常识」来补全 — 这就是幻觉（hallucination）的来源之一。

### 2. 单次 Prompt 难以驾驭复杂任务

如果一个任务涉及多个文件、多个步骤、多种角色配合，单靠一个 Prompt 很难控制住。模型容易漏步骤、忘约束、前后矛盾。**Prompt 最适合「单点、明确、少步骤」的任务**，不适合端到端的长链路。

### 3. Prompt 无法替代系统化流程

比如一个完整的开发流程：需求分析 → 架构设计 → 编码 → 测试 → 部署。如果只靠 Prompt 来推进，每个环节之间缺乏稳定的衔接，整体质量很难保证。

这正是更高级的 AI 协作方式出现的原因：

| 能力 | 说明 |
|------|------|
| **Rules** | 持久化的行为规范，确保 AI 每次都遵守特定规则 |
| **Skills** | 可复用的技能包，把多步骤任务封装成标准流程 |
| **Agent** | 能自主规划、调用工具、多步推理的智能体 |
| **MCP** | 让 AI 连接外部工具和数据源的标准协议 |

**Prompt 是 AI 协作的起点，但不是终点。** 当任务复杂到一定程度时，你需要从「写一句指令」升级到「配置一套系统」。后续文章会逐一展开这些进阶概念。
