# Rules

> 用好 AI

Rules（规范）是你给 AI 设定的长期行为约束 — 不需要每次重复，自动生效，持续遵守。

---

## 一、为什么需要 Rules

### 1. 一个你一定遇到过的场景

用 AI 写代码一段时间后，你会发现自己在 Prompt 里反复写同样的话：

> - 「用 Swift，不要用 OC」
> - 「UI 用 SnapKit 布局，不要用 Storyboard」
> - 「变量命名用 camelCase」
> - 「不要用 print，用 BTPrint」
> - 「提交信息用中文」

每次新对话都要重复一遍，漏了一条 AI 就按自己的「默认习惯」来。团队协作更麻烦 — 你的 Prompt 有这些约束，同事的没有，同一个项目里代码风格完全不统一。

解决思路很自然：**把反复出现的约束从 Prompt 中抽离出来，写成独立的规范文件，让 AI 每次自动加载。** 这就是 Rules。

### 2. Rules 的定位

在 AI 工具的协作体系里，Rules 和 Prompt、Skills 各司其职：

| 概念 | 回答的问题 | 生效方式 | 类比 |
|------|-----------|---------|------|
| **Prompt** | 这次做什么？ | 一次性，用完即消 | 你在餐厅点的那道菜 |
| **Rules** | 始终遵守什么？ | 自动加载，持续生效 | 厨房的卫生规范和操作标准 |
| **Skills** | 这类事按什么流程做？ | 按需触发，可复用 | 大厨的拿手菜谱 |

Rules 就像公司的编码规范，不管你今天写什么功能，这些规矩都要遵守。区别是：以前的编码规范写在文档里，靠人记、靠 Review 抓；现在的 Rules 写在配置文件里，**AI 自动遵守**。

---

## 二、Rules 能管什么

Rules 可以约束 AI 行为的方方面面，但最常用的是以下几类：

### 1. 编码风格

最基础也最高频的规则 — 统一团队的代码风格。

> - 缩进：4 空格
> - 命名：变量和函数 camelCase，类型和协议 PascalCase
> - 访问控制：非公开的属性和方法必须标记 `private`
> - 优先使用 `let` 而不是 `var`
> - 可空类型必须明确标注 `?`

### 2. 架构约束

确保 AI 生成的代码符合项目的架构设计。

> - ViewController 不直接发起网络请求，必须通过 Service 层
> - 页面文件放在 `Features/<Module>/` 目录下
> - 公共组件放在 `Components/` 目录下
> - 数据模型必须独立文件，不要写在 ViewController 里

### 3. 安全底线

这是最不能妥协的规则。

> - 禁止将密钥、密码、Token 硬编码到代码中
> - 所有用户输入必须做校验和净化
> - 敏感数据使用 Keychain 存储，禁止存 UserDefaults
> - 网络请求必须使用 HTTPS

### 4. 输出格式与行为偏好

控制 AI 回答的方式和风格。

> - 用中文回答
> - 不要在回答末尾总结你做了什么
> - 修改代码时只展示改动部分，不要重复输出整个文件
> - 提交信息用中文，不超过 50 字

### 5. 禁止行为

明确告诉 AI 什么不能做。

> - 不要生成 mock 数据
> - 不要添加注释到你没有修改的代码
> - 不要自行引入新的第三方依赖
> - 不要删除现有的注释或文档

## 三、Rules 的两个层次

Rules 根据作用范围，分为两个层次：

### 1. 通用规范：跨项目的个人偏好

这些规则和具体项目无关，是你个人（或团队）在所有项目中都希望 AI 遵守的。

```markdown
# 通用规范

## 语言与风格
- 用中文回答
- 回答简洁直接，不要废话
- 不要在回答末尾总结

## 代码习惯
- 优先使用 let，减少 var
- 错误处理要有具体的错误信息，不要只写 catch {}
- 提交信息用中文，格式：类型(范围): 描述
```

这类规则通常配置在全局位置，对所有项目生效。

### 2. 项目规范：仅当前仓库生效

这些规则和特定项目的技术栈、架构设计、团队约定绑定。

```markdown
# 项目规范

## 项目信息
- 技术栈：Swift 5.9+ / iOS 18.0
- UI 框架：UIKit + SnapKit
- 架构模式：MVC
- 包管理：Swift Package Manager (SPM)

## 目录约定
- 页面文件：Features/<Module>/
- 公共组件：Components/
- 网络层：Services/
- 数据模型：Models/

## 架构规则
- UI 布局使用 SnapKit，禁止 Storyboard / IB
- ViewController 不直接发起网络请求，通过 Service 层调用
- Model 必须独立文件，不要写在 ViewController 中
- 优先使用 delegate 模式，避免跨 VC 使用 Notification

## 常用命令
- 构建：Cmd + B
- 运行测试：Cmd + U
- 代码格式化：swiftformat .
```

这类规则放在项目根目录，和代码一起提交到 Git，团队共享。

### 两者的关系

| | 通用规范 | 项目规范 |
|---|---|---|
| **作用范围** | 所有项目 | 仅当前项目 |
| **内容** | 个人偏好、通用习惯 | 技术栈、架构、目录、命令 |
| **存储位置** | 全局配置目录 | 项目根目录 |
| **优先级** | 低 | 高（覆盖通用规范的同名规则） |

当两者冲突时，**项目规范优先**。比如你的通用规范写了「用双引号」，但项目规范写了「用单引号」，那在这个项目里就用单引号。

## 四、如何写好 Rules

Rules 不是写得越多越好。写得模糊，AI 不遵守；写得太多，AI 记不住。关键是：**每条规则都具体、可验证、有理由**。

### 原则一：写具体、可验证的指令

AI 对模糊的指令会「选择性遵守」— 有时候听，有时候不听。具体到可以检查「是否违反」的指令，遵守率最高。

| 写法 | 效果 |
|------|------|
| ❌ 「代码要简洁」 | AI 不知道什么算简洁，每次理解不同 |
| ✅ 「函数不超过 30 行，单个文件不超过 300 行」 | 清晰可量化，AI 能严格执行 |
| ❌ 「注释要写好」 | 什么叫「好」？ |
| ✅ 「所有 public 方法必须有 `///` 文档注释，包含参数和返回值说明」 | 明确到可以逐条检查 |
| ❌ 「不要写不安全的代码」 | 太抽象 |
| ✅ 「禁止将 API Key 硬编码到代码中，使用环境变量」 | 一目了然 |

### 原则二：给正反示例

Rules 里的示例比文字描述有效得多。AI 看到具体的代码，比理解抽象的规则更准确。

**错误处理规范**

✅ 正确：

```swift
func getUser(by id: String) throws -> User? {
    do {
        let user = try userService.fetch(id: id)
        return user
    } catch let error as NetworkError {
        Logger.warning("User not found", context: ["id": id])
        return nil
    } catch {
        Logger.error("Failed to get user", context: ["id": id, "error": error])
        throw DatabaseError.queryFailed(cause: error)
    }
}
```

❌ 错误：

```swift
func getUser(by id: String) -> User? {
    do {
        return try userService.fetch(id: id)
    } catch {
        print(error)  // 只打印，不处理
        return nil     // 吞掉异常
    }
}
```

### 原则三：说明「为什么」

AI 理解了规则背后的原因，在遇到边界情况时能做出更合理的判断，而不是机械地套用。

**优先使用 `let` 而不是 `var`**

> **原因**：
> - 不可变变量减少认知负担，阅读代码时不用追踪变量在哪里被改了
> - 防止意外修改引入 bug
> - 编译器可以做更好的优化
>
> **例外**：循环计数器、累加器等确实需要修改的场景可以用 `var`

### 原则四：设置优先级

不是所有规则的重要程度都一样。区分优先级，让 AI 知道什么必须遵守、什么尽量遵守。

**P0（必须遵守，违反则不可接受）**

> - 禁止硬编码密钥和密码
> - 所有用户输入必须校验
> - 提交前必须通过编译，无警告

**P1（强烈建议，Code Review 会提出）**

> - public 方法必须有 `///` 文档注释
> - 复杂业务逻辑必须有注释说明「为什么」
> - 单元测试覆盖核心逻辑

**P2（建议，不强制）**

> - 优先使用值类型（struct）而非引用类型（class）
> - 变量声明就近原则

### 常见误区

| 误区 | 问题 | 改进 |
|------|------|------|
| 规则太模糊 | AI 理解不一致，时遵守时不遵守 | 写到可以逐条检查的粒度 |
| 规则太多 | 上下文窗口有限，后面的规则容易被忽略 | 控制在核心规则 20-30 条以内，按优先级排列 |
| 规则和 Skills 混用 | Rules 里写了执行流程，导致每次任务都走同一个流程 | Rules 只写约束，流程放 Skills |
| 只写「不要做什么」| AI 知道不该做什么，但不知道该怎么做 | 正反示例搭配，既说禁止什么，也说推荐什么 |
| 长期不更新 | 项目技术栈变了，规则还是旧的 | 建立更新机制，发现问题及时补充 |

## 五、Rules 在主流 AI 工具中的配置

Rules 是一个通用概念，各 AI 工具都支持，只是**文件名和存放位置不同**。本质是同一件事：把规则写在约定位置的文件里，工具启动时自动加载。

### 同一概念，不同载体

| AI 工具 | 项目规范文件 | 全局规范位置 |
|---------|------------|------------|
| **Claude Code** | `CLAUDE.md` + `.claude/rules/*.md` | `~/.claude/CLAUDE.md` + `~/.claude/rules/*.md` |
| **Cursor** | `.cursor/rules/*.mdc` 或 `.cursorrules` | 设置中的 User Rules |
| **Trae** | `.trae/rules/` 或项目根目录 | 设置中配置 |
| **Windsurf** | `.windsurfrules` | 设置中配置 |
| **GitHub Copilot** | `.github/copilot-instructions.md` | VS Code 设置 |

### 为什么文件名不统一？

**目前行业没有统一标准，各厂商各自命名。** 就像 Node 用 `package.json`、Python 用 `pyproject.toml` — 概念相同，载体不同。`CLAUDE.md` 是 Claude Code 的约定，`.cursorrules` 是 Cursor 的约定，`AGENTS.md` 是一些团队自己约定的通用名称。本质都是同一个东西。

### 多工具共存：单一数据源

如果团队同时用 Claude Code 和 Cursor，维护两份规则容易不同步。推荐做法：**选一个文件作为主规范（如 `AGENTS.md`），其他文件只做跳转引用。**

```markdown
# CLAUDE.md

> 本文件仅用于兼容 Claude Code 的默认读取机制

本项目的 AI 协作规范统一维护在 [AGENTS.md](./AGENTS.md)。
- ❌ 不要在本文件中新增或修改规范
- ✅ 如需更新规则，请直接修改 AGENTS.md
```

`.cursorrules` 同理。这样只维护一份规则，改一处全局生效。

### 优先级机制

大多数工具的优先级逻辑一致：

```
企业/管理员策略（最低）
    ↓
全局个人规范
    ↓
项目根目录规范
    ↓
子目录规范（最高）
```

**离你当前工作的文件越近，优先级越高。** 比如你在 `src/frontend/` 目录下工作，如果这个目录有自己的规则文件，它会覆盖根目录的同名规则。

### 推荐的项目目录结构

不管用什么工具，规则文件都建议拆分成多个小文件，按主题组织：

```
your-project/
├── CLAUDE.md              # 项目基本信息（技术栈、常用命令）
└── .claude/
    └── rules/
        ├── coding-style.md    # 编码风格
        ├── architecture.md    # 架构约束
        ├── security.md        # 安全底线
        └── testing.md         # 测试规范
```

拆分的好处：

- **职责单一**：每个文件只管一个主题，修改时不会误伤其他规则
- **便于审查**：PR 里改了 `security.md`，Reviewer 一眼就知道改了安全规则
- **按需加载**：部分工具支持根据文件类型匹配规则，比如只在写测试文件时加载测试规范

## 六、Rules 的工程化管理

Rules 不是写一次就完事了。和代码一样，它需要版本管理、团队协作、持续迭代。

### 1. 纳入 Git 版本管理

把 Rules 文件和代码一起提交。团队成员拉取代码后自动获得最新规则，也可以通过 `git log` 追溯变更历史、通过 PR 审查修改。

### 2. 复利模式：发现问题 → 更新规则 → 全员受益

**每一次踩坑都能变成团队的积累。** 典型循环：

1. **发现问题**：Code Review 时发现 AI 生成的代码里用了 `print()` 而不是项目的 `BTPrint`
2. **分析根因**：规则里没有明确禁止 `print()`
3. **更新规则**：在 `coding-style.md` 里加一条：「禁止使用 print()，统一使用项目的 BTPrint 模块」
4. **提交同步**：`git commit -m "docs: prohibit print(), use BTPrint instead"`
5. **全员受益**：从此所有人的 AI 都不会再生成 `print()`

```
第 1 周：Code Review 发现 10 个规范问题
第 2 周：7 个问题（之前的 3 个已经加入规则，不再出现）
第 3 周：5 个问题
  ...
第 N 周：0 ~ 1 个问题（规则已经覆盖了常见场景）
```

前期投入的每一条规则，后续都在持续减少重复问题。

### 3. 定期回顾

建议每月检查一次：有没有过时的、缺失的、矛盾的、太模糊的规则。技术栈在变，Rules 也要跟着演进。

## 七、小结

| 维度 | 要点 |
|------|------|
| **Rules 是什么** | 长期生效的行为约束，自动加载，不需要每次重复 |
| **和 Prompt 的区别** | Prompt 管「这次做什么」，Rules 管「始终遵守什么」 |
| **两个层次** | 通用规范（跨项目个人偏好）+ 项目规范（当前仓库的技术栈和架构约束） |
| **写好的关键** | 具体可验证 + 正反示例 + 说明原因 + 分优先级 |
| **工程化管理** | 纳入 Git 版本管理 + 复利模式持续迭代 + 定期回顾 |
| **核心心态** | Rules 不是一次性的文档，而是团队知识的持续积累 |

**一句话总结：Prompt 让 AI 知道「做什么」，Rules 让 AI 知道「怎么做才对」。好的 Rules 不是约束 AI 的枷锁，而是让 AI 成为越来越靠谱的队友的训练手册。**
