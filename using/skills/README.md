# Skills

> 用好 AI

Skills（技能）是封装好的 AI 能力模块 — 把特定任务的流程、知识和工具打包在一起，让 AI 按照专业标准稳定执行。

如果说 [Prompt](../prompt/) 是「临时任务」，[Rules](../rules/) 是「行为规范」，那 Skills 就是「标准操作流程（SOP）」。

---

## 一、为什么需要 Skills

### 1. Prompt 的天花板

用 Prompt 完成简单任务没问题，但遇到复杂任务时，你会发现自己每次都在重复写一大段指令：

> 「帮我做 Code Review，要检查线程安全、循环引用、内存泄漏、架构规范、命名规范、错误处理，按严重程度分级输出，给出修复建议和代码示例……」

这段话写一次可以，但如果你每周都做 Code Review，每次都要写这么长，既麻烦又容易遗漏。而且 Prompt 一次性用完就消失了，下次还得重新写。

### 2. Skills 解决什么问题

Skills 把这类「反复执行的复杂任务」封装成一个可复用的模块。封装之后：

- **一句话触发**：说「帮我做 Code Review」，AI 自动按完整流程执行
- **质量稳定**：每次都按同一套标准检查，不会遗漏
- **可迭代**：发现流程有改进空间，更新 Skill 文件即可，所有人受益

### 3. Prompt、Rules、Skills 的关系

| 概念 | 回答的问题 | 生命周期 | 类比 |
|------|-----------|---------|------|
| **Prompt** | 这次做什么？ | 一次性 | 口头吩咐一件事 |
| **Rules** | 始终遵守什么？ | 长期生效 | 公司规章制度 |
| **Skills** | 这类事按什么流程做？ | 按需触发，可复用 | 标准操作手册（SOP） |

三者的协作方式：**Prompt 触发任务 → Skills 提供执行流程 → Rules 约束整个过程中的行为规范。**

从 iOS 开发者的视角来类比：Skills 很像 Framework / SDK。我们常用的 SDWebImage、Alamofire、SnapKit 都是封装好的能力模块；Skills 则是封装好的「AI 执行流程」— 一次构建，反复调用。

## 二、Skills 的结构

一个 Skill 本质上就是一个文件夹，核心是一个 Markdown 文件（通常叫 `SKILL.md`），外加可选的辅助资源。

### 基本目录结构

```
my-skill/
├── SKILL.md              # 核心文件（必须）
├── scripts/              # 可执行脚本（可选）
├── references/           # 参考文档（可选）
└── assets/               # 静态资源（可选）
```

### SKILL.md 的组成

SKILL.md 由两部分构成：**元数据（YAML Front Matter）** 和 **正文（Markdown）**。

```yaml
---
name: code-review              # 唯一标识
description: |                 # 触发描述：AI 据此判断何时激活
  当用户要求代码审查、Review 代码、检查代码质量时使用。
---

# 正文：给 AI 的操作指南

## 执行流程
1. 查看改动范围
2. 检查架构和设计
3. 检查安全问题
4. 检查代码质量
5. 输出结构化报告

## 输出格式
按严重程度分级：P0（必须修复）/ P1（建议修复）/ P2（可选优化）

## 禁止行为
- 不要跳过安全检查
- 不要只说"代码没问题"而不给具体分析
```

### 五个核心要素

| 要素 | 作用 | 在哪里写 |
|------|------|---------|
| **元数据** | name + description，告诉 AI 这个 Skill 是什么、什么时候触发 | YAML Front Matter |
| **适用上下文** | 这个 Skill 适用于什么场景、有什么前提条件 | 正文开头 |
| **执行流程** | 一步步怎么做，是 Skill 的核心 | 正文主体 |
| **约束规则** | 禁止行为、必须遵守的原则 | 正文中单独一节 |
| **输出规范** | 结果用什么格式呈现 | 正文中单独一节 |

### 三类可选资源

| 目录 | 用途 | 示例 |
|------|------|------|
| **scripts/** | 可直接执行的脚本，不占用上下文 | 日志分析脚本、格式化工具 |
| **references/** | 参考文档，AI 需要时按需加载 | API 规范、检查清单、编码规范 |
| **assets/** | 静态资源和模板 | 输出报告模板、checklist 模板 |

**为什么要拆分成多个文件？** 因为上下文窗口是有限的。如果把所有内容都塞进 SKILL.md，会浪费大量 Token。拆分后，AI 只在需要时加载对应文件，用完就可以释放。这就是「按需加载」的设计。

---

## 三、Skills 的加载机制

Skills 采用三级加载，按需逐步展开：

| 级别 | 加载内容 | 何时加载 | Token 消耗 |
|------|---------|---------|-----------|
| **L1** | name + description | AI 启动时始终加载 | 约 100 词 |
| **L2** | SKILL.md 完整正文 | Skill 被触发后 | < 5000 词 |
| **L3** | scripts / references / assets | AI 判断需要时 | 不限 |

**实际效果**：AI 随时知道有哪些 Skills 可用（L1），但只有真正需要时才读取详细流程（L2）和参考资料（L3）。这样既不浪费上下文，又能在需要时获得完整的专业知识。

这也解释了为什么 `description` 要写好 — 它是 AI 判断「是否触发这个 Skill」的唯一依据。description 写得模糊，Skill 就可能在该触发时没触发，或者不该触发时乱触发。

## 四、如何创建 Skills

### 方式一：手动编写

适合对流程有清晰想法的场景。直接创建 `SKILL.md` 文件，按上面的结构填入内容。

### 方式二：用 AI 辅助生成

更推荐的方式。直接告诉 AI 你的需求，让它帮你生成 Skill 的初稿，然后你再根据实际使用效果迭代优化。

> 「帮我创建一个 iOS Code Review 的 Skill，需要检查 Swift API 设计规范、循环引用、线程安全、架构规范、性能问题，按严重程度分级输出。」

AI 会生成完整的 SKILL.md 骨架，你在这个基础上调整细节，比手写快得多。

Anthropic 官方提供了一个叫 `skill-creator` 的工具（本身也是一个 Skill），专门用来生成其他 Skill。用它可以在几分钟内搭出一个高质量的 Skill 初稿。

### 创建流程

不管用哪种方式，完整的创建流程是：

1. **明确场景**：这个 Skill 解决什么问题？用户会怎么触发它？
2. **设计流程**：拆分成具体步骤，每步做什么、产出什么
3. **编写内容**：填充 SKILL.md 和必要的参考文档
4. **实际测试**：用真实任务测试，观察 AI 的执行效果
5. **迭代优化**：根据测试结果调整流程、补充约束

## 五、Skills 的存储与安装

### 存储位置

```
~/.claude/skills/           # 全局（所有项目可用）
<项目根目录>/.claude/skills/  # 项目级（仅当前项目可用）
```

全局 Skills 适合放通用能力（如 Code Review、Git 提交规范），项目 Skills 适合放和特定项目绑定的能力（如项目专属的部署流程）。

### 跨工具兼容

和 Rules 一样，Skills 的概念是通用的，各工具的存储位置略有不同：

| 工具 | Skills 目录 |
|------|------------|
| Claude Code | `.claude/skills/` |
| Cursor | `.cursor/skills/` |
| Trae | `.trae/skills/` |

部分工具会自动读取其他工具的 Skills 目录（比如 Cursor 也会读 `.claude/skills/`），所以 Skills 的跨工具复用通常不需要额外操作。

### 社区资源

不需要所有 Skill 都自己写。社区已经有大量现成的高质量 Skills：

| 资源 | 说明 |
|------|------|
| [skills.sh](https://skills.sh/) | 社区精选 Skill 集合，按领域分类，支持一键安装 |
| [Anthropic 官方 Skills](https://github.com/anthropics/skills) | 官方维护，是学习 Skill 编写的最佳参考 |

## 六、实战示例

### 示例 1：iOS Code Review Skill

```yaml
---
name: ios-code-review
description: |
  当用户要求审查 iOS/Swift 代码时使用。
  覆盖架构规范、内存安全、线程安全、性能、代码质量。
---

# iOS Code Review

## 执行流程

### 第一步：了解改动范围
查看 git diff，确认改动涉及哪些文件和模块。超过 500 行的改动按模块分批 review。

### 第二步：架构检查
- ViewController 是否承担了过多职责
- 网络请求是否通过 Service 层发起
- Model 是否独立定义，未混在 VC 中
- 是否符合项目的 MVC / MVVM 分层

### 第三步：内存与线程安全
- 闭包是否存在循环引用（检查 [weak self] / [unowned self]）
- delegate 是否声明为 weak
- 是否存在主线程外的 UI 操作
- 异步回调中是否正确处理了生命周期

### 第四步：代码质量
- 错误处理是否完善（不要只 catch {} 不处理）
- 是否有强制解包（`!`）的滥用
- 命名是否清晰、符合 Swift API Design Guidelines
- 是否有重复代码可以提取

### 第五步：输出报告

按严重程度分级：

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| P0 | 严重问题 | 必须修复才能合并 |
| P1 | 重要问题 | 建议合并前修复 |
| P2 | 改进建议 | 可以建 follow-up |

每个问题附上：文件名 + 行号 + 问题描述 + 修复建议（含代码）。

## 禁止行为
- 不要只说"代码没问题"而不给分析
- 不要只指出问题而不提供修复代码
- 不要遗漏循环引用和线程安全检查
```

### 示例 3：TDD Skill（测试驱动开发）

```yaml
---
name: test-driven-development
description: |
  当用户要求实现任何功能、函数或模块时使用。
  强制按照测试驱动开发流程：先写测试，再写实现。
---

# 测试驱动开发（TDD）

## 核心原则

红-绿-重构（Red-Green-Refactor）循环：

1. **红**：先写一个会失败的测试
2. **绿**：写最少的代码让测试通过
3. **重构**：优化代码，保持测试通过

## 执行流程

### 第一步：编写测试（必须先完成）

在写任何实现代码之前：
1. 创建测试文件
2. 编写至少一个测试用例
3. 向用户展示测试代码
4. 确认后才能编写实现

### 第二步：编写实现

写最少的代码让测试通过。不要过度设计。

### 第三步：重构

测试全部通过后，优化代码结构，保持测试绿色。

## 禁止行为

- ❌ 先写实现代码再补测试
- ❌ 跳过测试直接给实现
- ❌ 写没有断言的测试
```

## 七、写好 Skills 的关键原则

### 1. description 决定触发效果

description 是 AI 判断「是否使用这个 Skill」的唯一依据。写得清晰全面，Skill 才能在正确的时机被触发。

```yaml
# ❌ 太简略，AI 不知道什么时候该触发
description: 做代码审查

# ✅ 清晰描述触发场景
description: |
  当用户要求审查代码、review 代码、检查代码质量时使用。
  覆盖安全性、性能、可维护性检查。
```

### 2. 流程要有节奏感

不要把所有检查点平铺在一起。按照从宏观到微观的顺序编排：先看整体，再看细节。

```
✅ 了解改动范围 → 架构检查 → 安全检查 → 代码质量 → 输出报告
❌ 安全、命名、架构、性能、格式... 全部混在一起
```

### 3. 正文精简，细节外置

SKILL.md 只放核心流程，控制在 500 行以内。详细的检查清单、API 文档等放在 `references/` 目录，正文中用引用指向：

```markdown
## 安全检查
按照 references/security-checklist.md 逐项检查。
```

### 4. 提供正反示例

和写 Rules 一样，代码示例比文字描述有效得多。在关键步骤给出「正确做法」和「错误做法」的对比。

### 5. 先确认，再执行

对于会修改代码的操作，让 AI 先展示计划或报告，等用户确认后再执行。避免 AI「自作主张」带来不可控的改动。

## 八、Skills 与其他概念的关系

最后把 Skills 放到整个 AI 协作体系中，看看它和其他概念的关系：

| 概念 | 本质 | 回答的问题 |
|------|------|-----------|
| **Prompt** | 一次性指令 | 这次做什么？ |
| **Rules** | 持久化的约束 | 始终遵守什么？ |
| **Skills** | 可复用的流程 | 这类事怎么做？ |
| **MCP** | 外部连接协议 | 用什么工具做？ |
| **Agent** | 自主执行者 | 谁来做？ |

它们的协作方式：

```
用户说「帮我做 Code Review」（Prompt）
    ↓
AI 匹配到 Code Review Skill，按流程执行（Skills）
    ↓
执行过程中遵守项目编码规范（Rules）
    ↓
需要读取代码时通过文件系统工具获取（MCP）
    ↓
整个过程由 AI Agent 自主编排和推进（Agent）
```

**Skills 是这个体系中的「专业能力层」**— 它让 AI 从「什么都能聊两句但不深入」变成「在特定领域能按专业标准稳定输出」。
