# Projects

> 用好 AI

Projects（项目空间）解决的是 AI 的上下文记忆问题 — 让 AI 从「每次对话都从零开始」变成「持续理解你的项目全貌」。

前面几篇讲的 [Prompt](../prompt/)、[Rules](../rules/)、[Skills](../skills/) 都是告诉 AI「做什么、怎么做」。但还有一个根本问题没解决：**AI 不认识你的项目。**

你让它写代码，它不知道你的技术栈；你让它改 Bug，它不知道你的架构；你让它设计方案，它不知道你的业务背景。每次都要把这些信息重新喂一遍，效率很低。

Projects 就是来解决这个问题的：**给 AI 一个持久的工作空间，让它能持续积累和使用项目相关的信息。**

---

## 一、为什么需要 Projects

### 普通对话的局限

AI 的普通对话有两个硬伤：

**记忆是短暂的** — 对话结束后，AI 忘掉一切。下次你再聊同一个项目，它又是一张白纸。即使在同一次对话中，当内容超过上下文窗口（128K ~ 200K Token），早期的信息也会被截断。

**上下文是割裂的** — 你和 AI 聊了需求，再聊架构，再聊代码。但这三次对话之间没有关联，AI 在写代码时不知道你之前聊过的架构决策。

这在简单任务中不是问题。但真实项目是一个持续推进的过程：理解需求 → 设计架构 → 编写代码 → 修复 Bug → 迭代优化。如果 AI 每次都从零开始，它就只能当一个「一次性问答机器」，而不是一个「持续参与的项目成员」。

### Projects 解决什么

Projects 让 AI 拥有一个和你的项目绑定的工作空间。在这个空间里：

- **信息持久化**：项目文档、代码、设计稿、对话历史都可以保存，AI 随时调用
- **上下文连续**：多次对话之间共享项目信息，AI 能基于之前的讨论继续推进
- **知识积累**：随着项目推进，AI 对项目的理解越来越深，输出质量越来越高

一句话：**Projects 让 AI 从「聊天工具」升级为「工作系统」。**

---

## 二、Projects 的核心能力

一个 AI 项目空间通常能保存以下几类信息：

| 类型 | 作用 | 示例 |
|------|------|------|
| **项目背景** | 让 AI 理解项目是什么、做什么 | 产品简介、业务背景、目标用户 |
| **文档资料** | 提供专业知识和参考依据 | PRD、API 文档、设计规范 |
| **代码和文件** | 让 AI 基于真实代码工作 | 源代码、配置文件、数据库 Schema |
| **规则和约束** | 约束 AI 的行为 | 编码规范、架构约束、安全要求 |
| **历史对话** | 让 AI 记住之前的讨论和决策 | 架构讨论记录、需求确认记录 |

这些信息组合在一起，构成了 AI 对你项目的「认知」。信息越完整，AI 的输出就越贴合你的实际需求。

## 三、Projects 在不同工具中的形态

「Projects」不是某一个工具的专属功能，而是一个通用概念。不同 AI 工具用不同的方式实现它：

### Claude.ai Projects

Claude 的 Web 端提供了显式的 Projects 功能。你可以创建一个项目，上传文档和文件，设置项目级的系统指令。在项目内的所有对话都共享这些信息。

**适合场景**：非编程类的项目协作，如产品设计、内容创作、研究分析。

### Claude Code / Cursor 的项目上下文

在 AI 编程工具中，Projects 的概念更隐式 — 它就是你的代码仓库本身。

当你在一个代码仓库中使用 Claude Code 或 Cursor 时，AI 会自动获取项目上下文：

- **代码文件**：AI 可以读取、搜索、修改项目中的代码
- **CLAUDE.md / .cursorrules**：项目级的规则和背景信息（就是前面讲的 [Rules](../rules/)）
- **Git 历史**：了解最近的改动和提交记录
- **目录结构**：理解项目的组织方式

**适合场景**：编程类任务，AI 直接在你的代码仓库中工作。

### 对比

| | Claude.ai Projects | Claude Code / Cursor |
|---|---|---|
| **上下文来源** | 手动上传文档和文件 | 自动读取代码仓库 |
| **信息持久化** | 云端存储，跨对话保留 | 项目文件本身就是持久化的 |
| **适合场景** | 产品、设计、研究等非编程协作 | 编程、代码审查、架构设计 |
| **协作方式** | 在 Web 对话中交流 | 在终端/IDE 中直接操作代码 |

本质是一样的：**都是给 AI 提供一个持久的、项目相关的上下文环境。**

## 四、如何组织一个 AI 友好的项目

不管用什么工具，项目的信息组织方式直接影响 AI 的理解效果。以下是让 AI 更好理解项目的实践建议。

### 1. 提供清晰的项目入口

AI 接触你的项目的第一步，是读到一个「项目概述」。在 Claude.ai 中是项目描述；在代码仓库中是 `CLAUDE.md` 或 `README.md`。

一个好的项目入口应该包含：

```markdown
# 项目名称

## 基本信息
- 技术栈：Swift 5.9+ / iOS 18 / UIKit + SnapKit
- 架构：MVC
- 包管理：SPM

## 目录结构
- Features/：业务模块（按功能划分）
- Components/：公共 UI 组件
- Services/：网络层和数据层
- Models/：数据模型

## 常用命令
- 构建：Cmd + B
- 测试：Cmd + U
```

AI 读到这些信息后，就知道你的项目用什么技术栈、代码放在哪、怎么构建。后续所有对话都基于这个理解展开。

### 2. 按需提供上下文，而非一股脑全塞

上下文窗口是有限的。把所有文档都扔给 AI，它反而抓不住重点 — 无关信息会分散模型的注意力（attention），降低输出质量。这就是上下文工程（Context Engineering）的核心原则：**给最小但充分的信息。**

就像设计函数接口一样 — 对外只暴露必要的参数，隐藏内部实现。给 AI 的上下文也应该是完成任务所需的「接口面」，而不是把实现细节一股脑倒进去。

> 让 AI 修一个网络请求的 Bug → 给它网络层的代码 + 相关的 API 文档就够了，不需要给 UI 层的代码。

> 让 AI 设计一个新功能 → 给它 PRD + 现有的架构说明 + 相关模块的代码，不需要给测试用例。

### 3. 用多层索引组织信息

当项目变大时，一个扁平的 CLAUDE.md 不够用了。更好的做法是建立**多层索引**：

```
MyApp/
├── CLAUDE.md                      # 顶层：项目概述 + 技术栈 + 通用约束
├── Features/
│   ├── README.md                  # 模块层：各业务模块的职责和依赖
│   ├── Login/
│   │   └── README.md              # 子模块层：登录模块的具体实现说明
│   └── Home/
│       └── README.md
└── Services/
    └── README.md                  # 网络层的接口约定和使用方式
```

每一层 README 只描述「我是谁、我负责什么、我的约束是什么」，不重复上层已有的信息。AI 通过顶层索引快速定位到需要的模块，再深入读取该模块的详细信息。

这样做的好处：每次任务只需打包「顶层规则 + 当前模块 README + 任务描述」的最小集，而不是把整个项目塞进上下文。

### 4. 让文档和代码保持同步

AI 依赖你提供的信息来理解项目。如果文档是过时的，AI 的输出就会基于错误的理解。

几个关键文件需要保持更新：

- **项目入口文件**（CLAUDE.md）：技术栈变了要更新，目录结构改了要更新
- **API 文档**：接口变更后及时同步
- **架构说明**：重大架构调整后补充记录

一个实用的做法：在 Rules 中加一条「任务完成后，如果涉及新模块或架构变更，更新对应层级的 README」。这样文档会随项目演进自动维护，而不是靠人记着去更新。

### 5. 用 iOS 项目的视角理解

Projects 很像 Xcode 的 Project / Workspace。一个典型的 iOS 项目结构：

```
MyApp/
├── Features/
│   ├── Login/
│   ├── Home/
│   └── Profile/
├── Services/
├── Models/
├── Components/
└── Resources/
```

在 AI 的视角里，这个结构同样成立。AI 理解了目录约定，就能在正确的位置创建文件、在正确的模块中查找代码。

加上 `CLAUDE.md` 和 `.claude/rules/` 之后：

```
MyApp/
├── CLAUDE.md                  # 项目概述 + 常用命令
├── .claude/
│   └── rules/                 # 编码规范、架构约束
├── Features/
├── Services/
├── Models/
├── Components/
└── Resources/
```

这样 AI 就像一个「始终记得项目全貌」的团队成员 — 知道项目的背景、遵守团队的规范、了解代码的结构。

## 五、Projects 的使用策略

### 简单任务：不需要 Projects

如果只是问一个语法问题、写一段独立的工具函数、翻译一段文本，普通对话就够了，不需要搭建项目空间。

### 持续性项目：值得投入

当任务具备以下特征时，Projects 的价值就体现出来了：

- **跨多次对话**：不是一次就能完成的任务
- **需要项目背景**：AI 必须了解技术栈、架构、业务才能做好
- **涉及多个模块**：需要在不同文件和模块之间协调
- **会持续迭代**：需求变更、Bug 修复、功能扩展

### 投入产出比

Projects 的前期投入是「整理项目信息、编写 CLAUDE.md、配置 Rules」。这个投入不大，但收益是持续的 — 项目推进得越久，AI 对项目的理解越深，协作效率越高。

前面几篇讲的 Rules 和 Skills 也是 Projects 的一部分。**它们共同构成了 AI 对你项目的完整认知：Rules 定义行为规范，Skills 定义执行流程，Projects 提供项目背景和持久上下文。**

## 六、小结

| 维度 | 要点 |
|------|------|
| **Projects 是什么** | AI 的项目工作空间，解决上下文记忆问题 |
| **解决什么问题** | 让 AI 从「每次从零开始」变成「持续理解项目全貌」 |
| **核心能力** | 信息持久化 + 上下文连续 + 知识积累 |
| **不同工具的形态** | Claude.ai 的 Projects 功能、Claude Code/Cursor 的代码仓库上下文 |
| **关键实践** | 写好项目入口、按需提供上下文、保持文档同步 |
| **与其他概念的关系** | Projects 提供背景，Rules 约束行为，Skills 定义流程，三者共同构成 AI 的项目认知 |

**一句话总结：Prompt 管一次任务，Rules 管行为规范，Skills 管执行流程，Projects 管项目记忆。四者搭配，AI 才能从「一次性聊天工具」变成「持续参与的项目成员」。**
